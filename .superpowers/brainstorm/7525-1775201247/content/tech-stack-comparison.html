기술 스택 접근법 비교

영어 단어 카드 게임 "Dripple" — 모바일 + 웹, 실시간 멀티플레이, AI 대전

A

React Native + Expo (추천)

프론트엔드: React Native (Expo) + React Native Web

게임 렌더링: React Native Reanimated + Gesture Handler (카드 애니메이션)

백엔드: Supabase (Auth, DB, Realtime WebSocket)

AI: Claude API / OpenAI (문장 검증 + AI 대전)

광고: Google AdMob

장점

  • 하나의 코드베이스로 iOS + Android + Web 모두 커버
  • React 생태계의 풍부한 라이브러리
  • Expo로 빠른 프로토타이핑 & 빌드
  • Supabase Realtime으로 실시간 대전 구현 용이
  • gstack 워크플로우와 호환성 좋음

단점

  • 복잡한 카드 애니메이션에서 네이티브 대비 성능 제한
  • 게임 전용 엔진이 아니라서 파티클/이펙트 직접 구현 필요
B

Flutter + Flame Engine

프론트엔드: Flutter (Dart)

게임 렌더링: Flame Engine (2D 게임 엔진, Flutter 네이티브)

백엔드: Firebase (Auth, Firestore, Cloud Functions)

AI: Claude API / OpenAI

광고: Google AdMob (Flutter 플러그인)

장점

  • Flame Engine으로 카드 애니메이션/파티클 효과 강력
  • iOS + Android + Web 모두 지원
  • 네이티브급 렌더링 성능 (Skia 엔진)
  • 듀오링고 스타일 풍부한 애니메이션 구현에 적합

단점

  • Dart 학습 필요 (JavaScript/TypeScript와 다름)
  • 웹 빌드 성능이 모바일 대비 약함
  • gstack과의 호환성 낮음 (JS 생태계 아님)
C

Next.js + PixiJS (웹 퍼스트)

프론트엔드: Next.js (React) + Capacitor (모바일 래핑)

게임 렌더링: PixiJS (WebGL 2D 렌더링) + GSAP (애니메이션)

백엔드: Supabase 또는 자체 Node.js + WebSocket

AI: Claude API / OpenAI

광고: Google AdSense (웹) + AdMob (모바일)

장점

  • PixiJS로 게임급 2D 렌더링 (60fps 카드 애니메이션)
  • 웹 기술 기반이라 접근성 높음
  • SEO 지원 (Next.js SSR)
  • gstack 워크플로우 완벽 호환
  • Capacitor로 모바일 앱 패키징 가능

단점

  • Capacitor 래핑이 네이티브 앱 대비 UX 약간 떨어짐
  • 모바일 디바이스 API 접근 제한적
  • 앱스토어 심사에서 웹뷰 앱 거절 위험

💡 추천: Option A — React Native + Expo

카드 게임의 인터랙션(드래그, 스와이프, 부채꼴 배열)은 React Native Reanimated로 충분히 구현 가능하며, Expo의 빠른 빌드 파이프라인과 Supabase Realtime의 WebSocket 지원이 AI 대전 + 온라인 실시간 대전 모두를 효율적으로 커버합니다. gstack 워크플로우와도 잘 맞아 개발 생산성이 높습니다.
